# MiniBattleSimulator 战斗系统机制与配置说明文档

## 1. 系统架构概览

MiniBattleSimulator 采用**数据与逻辑分离（数据驱动）**架构模式。核心战斗逻辑独立于任何具体的游戏引擎表现层，可以在无图形界面的环境下独立运行和验证。

整个系统分为四个主要层级：
*   **核心层 (Core)**：位于 `core/` 目录。提供基础数学运算（`BattleMath`）、战斗公式（`BattleFormula`）、事件派发系统（`BattleEvent`）、Timeline 执行引擎（`SkillTimeline`）等基础能力。
*   **模块层 (Modules)**：位于 `modules/` 目录。实现战斗系统的核心业务逻辑，包括战斗主流程控制（`BattleMain`）、属性管理、技能与Buff系统、被动技能统一框架（`BattlePassiveSkill` + `PassiveHandlers`）、回合/行动顺序判定（`BattleActionOrder`）、伤害/治疗结算等。
*   **配置层 (Config)**：位于 `config/` 目录。由 JSON 数据表和 Lua 脚本组成，定义了角色属性、技能基础数值、Buff模板等静态数据，以及技能 Timeline 帧序列与被动触发定义（`config/passive/passive_defs.lua`）等动态行为配置。
*   **表现层 (UI/View)**：位于 `ui/` 目录。提供控制台渲染器（`ViewportRenderer`）用于打印战斗日志，并通过 `BattleVisualEvents` 事件订阅核心层派发的事件，驱动真实的动画和特效表现。

## 2. 战斗核心规则

### 2.1 战斗流程与行动顺序
战斗由 `BattleDriver` 驱动，在 `BattleMain` 中统筹控制整个生命周期：

1.  **初始化**：重置所有子系统，设定随机数种子，从配置加载敌我双方的角色面板属性、站位以及携带的技能。
2.  **行动判定 (Action Order)**：采用基于**速度 (SPEED)** 属性的排序机制，由 `BattleActionOrder` 动态决定下一个可行动的单位。
3.  **回合循环 (Round Execution)**：
    *   **回合开始**：结算持续伤害 (DOT)、持续治疗 (HOT) 等 Buff，触发"回合开始"时机的被动技能。
    *   **行动选择**：检查单位是否处于硬控状态（如眩晕、冰冻）。若无，则调用 `SelectAvailableSkill` 评估技能优先级、冷却和消耗，选定要释放的技能并确定目标。
    *   **技能执行**：通过 `CastSkillInSeq` 调用技能脚本的 `BuildTimeline` 生成逻辑帧序列，由 `SkillTimeline.Execute` 逐帧执行，派发 `SKILL_TIMELINE_FRAME` 等战斗事件。
    *   **回合结束**：结算 Buff 持续时长递减，触发"回合结束"时机的被动逻辑。
4.  **胜负判定**：
    *   一方阵容全灭，另一方获胜。
    *   达到最大回合数限制（由 `maxSteps` 决定，默认 20000），触发平局或防守方胜利。

### 2.2 属性体系与战斗公式

*   **核心属性**：
    *   基础属性：`HP` (生命), `ATK` (攻击), `DEF` (防御), `SPEED` (速度)
    *   进阶概率：`CRIT_RATE` (暴击率), `CRIT_DMG` (暴击伤害)
*   **伤害公式**：
    ```
    基础伤害 = (攻击者ATK - 防御者DEF) * 技能伤害倍率 / 10000
    最低伤害 = 1
    ```
*   **治疗公式**：
    ```
    治疗量 = 施法者ATK * 治疗倍率 / 10000
    ```

### 2.3 英雄属性成长

英雄属性从 `res_rgl_hero.json` 读取基础值，通过等级成长和星级乘数计算最终面板：

```lua
hpGrowthRate  = 0.06 + quality * 0.008
atkGrowthRate = 0.05 + quality * 0.006
defGrowthRate = 0.04 + quality * 0.005

starMultiplier = 1.0 + (star - 1) * 0.15   -- star=5 → 1.6

finalHp  = (baseHp + hpGrowth) * starMultiplier
finalAtk = (baseAtk + atkGrowth) * starMultiplier
finalDef = (baseDef + defGrowth) * starMultiplier
```

### 2.4 敌人属性成长

敌人属性按职业系数硬编码基础值，通过 quality/type/star 三重乘法叠加：

```lua
hpGrowthRate  = 0.12 + quality * 0.015
atkGrowthRate = 0.095 + quality * 0.012
defGrowthRate = 0.075 + quality * 0.010

qualityMultipliers = {1.0, 1.06, 1.12, 1.18, 1.26, 1.34}
typeMultipliers = {[0]=1.0, [1]=1.15, [2]=1.35}
starMultiplier = 1.0 + (star-1) * 0.15

totalMultiplier = qualityMultiplier * typeMultiplier * starMultiplier
atkMultiplier = 1.0 + (totalMultiplier - 1.0) * 0.45

hp  = (baseHp + hpGrowth) * totalMultiplier
atk = (baseAtk + atkGrowth) * atkMultiplier
def = (baseDef + defGrowth) * totalMultiplier
```

## 3. 技能系统

技能系统采用 **Timeline 执行体系**，所有技能脚本必须实现 `BuildTimeline` 函数。

### 3.1 技能类型

| Type | 名称 | CD | 能量消耗 | 说明 |
|------|------|-----|---------|------|
| 1 | 普攻 (NORMAL) | 无 | 0 | 每回合可用，回复20能量 |
| 2 | 主动 (ACTIVE) | 3回合 | 0 | CD 完成后可用 |
| 3 | 大招 (ULTIMATE) | 无 | 100 | 能量满时可用 |
| 4 | 被动 (PASSIVE) | - | - | 常驻生效 |

### 3.2 技能选择优先级

`SelectAvailableSkill` 按以下顺序选择：
1.  大招（ULTIMATE）— 能量 ≥ 100
2.  主动技能（ACTIVE）— CD = 0
3.  普攻（NORMAL）— 兜底

### 3.3 技能 ID 编码

```
actualSkillId = classId * 10 + level
```

禁止使用 `classId * 100 + level`（9位ID导致加载失败）。

### 3.4 Timeline 执行流程

```
CastSkillInSeq
  └─ LoadSkillLua(skillId)
  └─ skillScript.BuildTimeline(hero, targets, skill)
  └─ SkillTimeline.Execute(hero, targets, skill, timeline)
       ├─ 排序帧序列 (按 frame 升序)
       ├─ 逐帧执行 frame.execute(context, frameCopy)
       ├─ 每帧派发 SKILL_TIMELINE_FRAME 事件
       └─ 派发 SKILL_TIMELINE_COMPLETED 事件
```

### 3.5 配置三层架构

每个技能由三层配置共同定义，修改时必须同步更新：

| 层级 | 文件 | 关键参数 |
|------|------|----------|
| JSON 静态 | `config/res_skill_rgl.json` | SkillParam(倍率), Buff(持续/叠加), CoolDownR, Cost |
| Lua 脚本 | `config/skill_rgl/skill_{ID}.lua` | BuildTimeline 中的 damageRate, ApplyBuff 参数 |
| Buff 定义 | `config/buff/buff_{ID}.lua` | duration, canStack, maxStack, effects |

### 3.6 被动技能统一框架

当前被动技能不再使用 `event_*.lua + war_*.lua` 双文件模式，而是统一为：

| 层级 | 文件 | 职责 |
|------|------|------|
| 定义层 | `config/passive/passive_defs.lua` | 维护 `classId -> triggers` 的触发表 |
| 逻辑层 | `modules/passive_handlers.lua` | 实现具体被动逻辑 |
| 调度层 | `modules/battle_passive_skill.lua` | 注册、触发、读取 `hero.passiveRuntime` |

被动分为两类：
- **脚本型被动**：例如追击、格挡、战意、感染，直接在 handler 中执行逻辑
- **运行时状态型被动**：例如连击精通、寒冰亲和、雷电亲和，在 `BattleBegin` 写入 `hero.passiveRuntime`，由主动技能或通用结算逻辑读取

示例：
- `8000300` 连击精通写入 `comboMasterMinRate = 5000`
- `8000800` 寒冰亲和写入 `iceDamageBonusPct = 1000`
- `8000900` 雷电亲和写入 `thunderChainChanceBonus = 2000`

## 5. 视觉事件系统

逻辑层通过 `BattleEvent.Publish` 派发事件，表现层（Console/Web/Unity）通过订阅 `BattleVisualEvents` 事件来呈现战斗画面，实现逻辑与表现的完全解耦。

### 5.1 核心事件

| 事件 | 说明 |
|------|------|
| SKILL_TIMELINE_STARTED | Timeline 开始执行 |
| SKILL_TIMELINE_FRAME | 帧执行完成 |
| SKILL_TIMELINE_COMPLETED | Timeline 执行完成 |
| DAMAGE_DEALT | 伤害事件 |
| HEAL_RECEIVED | 治疗事件 |
| BUFF_ADDED / BUFF_REMOVED | Buff 变化 |
| HERO_STATE_CHANGED | 英雄状态变化 |
| HERO_DIED | 英雄阵亡 |
| TURN_STARTED / TURN_ENDED | 回合事件 |
| BATTLE_STARTED / BATTLE_ENDED | 战斗开始/结束 |
