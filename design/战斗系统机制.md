# MiniBattleSimulator 战斗系统机制与配置说明文档

## 1. 系统架构概览

MiniBattleSimulator 采用了典型的**数据与逻辑分离（数据驱动）**架构模式。核心战斗逻辑独立于任何具体的游戏引擎表现层，可以在无图形界面的环境下独立运行和验证。

整个系统分为四个主要层级：
*   **核心层 (Core)**：位于 `core/` 目录。提供基础数学运算（`BattleMath`）、战斗公式（`BattleFormula`）、事件派发系统以及战斗生命周期、技能加载执行（`SkillExecutor`）等基础能力。
*   **模块层 (Modules)**：位于 `modules/` 目录。实现战斗系统的核心业务逻辑，包括战斗主流程控制（`BattleMain`）、属性管理、技能与Buff系统、回合/行动顺序判定（`BattleActionOrder`）、伤害/治疗结算等。
*   **配置层 (Config)**：位于 `config/` 目录。由 JSON 数据表和 Lua 脚本组成，定义了角色属性、技能基础数值、Buff模板等静态数据，以及技能关键帧、自定义效果等动态行为脚本。
*   **表现层 (UI/View)**：位于 `ui/` 或其他外围系统。提供控制台渲染器（`ConsoleRenderer`）用于打印战斗日志，并通过 `BattleEvent` 监听核心层派发的事件，驱动真实的动画和特效表现。

## 2. 战斗核心规则

### 2.1 战斗流程与行动顺序
战斗由 `BattleDriver` 驱动，在 `BattleMain` 中统筹控制整个生命周期：

1.  **初始化**：重置所有子系统，设定随机数种子，从配置加载敌我双方的角色面板属性、站位以及携带的技能。
2.  **行动判定 (Action Order)**：采用类似“跑条”或基于**速度 (SPEED)** 属性的排序机制，由 `BattleActionOrder` 动态决定下一个可行动的单位。
3.  **回合循环 (Round Execution)**：
    *   **回合开始**：结算持续伤害 (DOT)、持续治疗 (HOT) 等 Buff，触发“回合开始”时机的被动技能。
    *   **行动选择**：检查单位是否处于硬控状态（如眩晕、冰冻）。若无，则调用 `SelectAvailableSkill` 评估技能优先级、冷却和消耗，选定要释放的技能并确定目标。
    *   **技能执行**：扣除能量或将技能进入冷却，执行技能具体的逻辑脚本，派发 `SkillCast` 等战斗事件。
    *   **回合结束**：结算 Buff 持续时长递减，触发“回合结束”时机的被动逻辑。
4.  **胜负判定**：
    *   一方阵容全灭，另一方获胜。
    *   达到最大回合数限制（由常量 `MAX_BATTLE_ROUNDS` 决定，默认通常为 100），触发平局或防守方胜利等特定规则。
    *   双方同归于尽判定为平局。

### 2.2 属性体系与战斗公式
底层数值计算高度依赖 `core/battle_formula.lua`，并统一使用**万分比（10000 = 100%）**作为倍率基础，避免浮点数精度丢失。

*   **属性来源与成长**：
    *   **基础面板**：英雄和敌人的基础属性（`AtkBaseNum`, `HpBaseNum` 等）分别配置在 `res_ally_info.json` 和 `enemy_data.lua` 中。
    *   **成长机制**：英雄依据 `res_ally_level.json` 随等级获得属性成长；敌人则依据职业系数（如坦克、输出）与等级差动态计算。
    *   **最终计算**：最终面板 = `(基础值 + 成长值) * 品质倍率 * 星级倍率`。
*   **核心属性**：
    *   基础属性：`HP` (生命), `ATK` (攻击), `DEF` (防御), `SPEED` (速度)。
    *   进阶概率：`CRIT_RATE` (暴击率), `CRIT_DMG` (暴击伤害), `HIT_RATE` (命中率), `DODGE_RATE` (闪避率), `RES` (效果抵抗)。
*   **伤害公式计算步骤**：
    1.  **基础伤害**：`(攻击者ATK - 防御者DEF) * 技能伤害倍率 / 10000`（设有最低伤害值兜底，通常为 1）。
    2.  **命中判定**：由攻击者的 `HIT_RATE` 和防御者的 `DODGE_RATE` 共同决定是否命中。
    3.  **暴击判定**：命中后根据 `CRIT_RATE` 判定，若暴击，基础伤害乘以暴击倍率（如默认 1.5 倍，PVP 可能调整）。
    4.  **格挡判定**：若触发格挡，进一步按比例减伤。
    5.  **最终伤害**：`基础伤害 * (1 + 最终伤害加成/10000) * (1 - 最终伤害减免/10000)`。
*   **治疗公式**：
    `施法者ATK * 治疗倍率 / 10000 * (1 + 治疗加成/10000)`，系统通常会在此基础上附加约 **±10%** 的随机数值波动以增加变数。

## 3. 技能系统配置与机制

技能系统采用**“静态数据（JSON）+ 动态行为（Lua）”**相分离的双层体系。

### 3.1 静态基础配置 (`res_skill.json`)
由数值策划配置，定义技能的基础面貌：
*   `ID` / `ClassID`: 技能唯一标识及分类原型。
*   `Type`: 技能类型。
    *   `1`: 普攻（无冷却、无消耗）。
    *   `2`: 主动技能（受冷却回合 `CoolDownR` 限制）。
    *   `3`: 大招/必杀技（受能量 `Cost` 限制）。
    *   `4` / `9`: 被动技能或触发技。
*   `CoolDownR`: 冷却回合数。
*   `Cost`: 释放技能消耗的能量。
*   `SkillParam`: 数值参数数组（万分比），通常 `SkillParam[0]` 为基础伤害倍率。
*   `Buff1` ~ `Buff5`: 附带的 Buff 数组（包含触发几率、目标、回合数、BuffID等）。

### 3.2 动态行为配置 (`skill_{ID}.lua`)
定义技能的具体表现与结算时机，主要分为两种模式：

#### 模式 A：时间轴驱动模式 (`actData`)
大部分主动技能使用基于关键帧的 `actData` 结构，与动画表现强绑定：
*   包含多个阶段（Phase），每个阶段定义 `atLeastTimeS`（最小持续时间）。
*   `keyFrameDatas`（关键帧数组）：定义在特定时间点（`TriggerS`）触发的具体逻辑。
    *   `datatype`: 事件类型。如 `DWCommon.DamageData`（触发伤害计算）、`DWCommon.LaunchBuff`（挂载 Buff）、`DWCommon.LaunchSpell`（释放子法术/多段攻击）。
    *   `data`: 字符串化的 Lua 表，通过 `load` 反序列化。例如 `"{ damageRate = 12000 }"`。倍率优先取此处的 `damageRate`，其次取 JSON 中的 `SkillParam`。

#### 模式 B：自定义脚本模式 (高级用法)
对于机制复杂的技能，核心层提供了强大的脚本干预能力。在技能脚本中，可以通过声明特定函数来接管默认逻辑：
*   **`CheckCondition(hero, skill, condition)`**：在技能释放前检查特殊条件。可用于判断当前回合数、友方死亡人数、或特定 Buff 层数（例如：`return BattleMain.GetCurrentRound() >= 3`）。
*   **`SelectTarget(hero, skill)`**：自定义目标筛选逻辑。可通过 `BattleScriptExp` 调用底层方法，如 `GetFriendSortByAttrId(owner, 9, false)`（选择攻击力最高的队友）。
*   **`Execute(hero, targets, skill)`**：一旦声明此函数，核心层将跳过 `actData` 解析，完全由脚本控制技能行为（如实现分段伤害、动态目标切换、复杂战场环境改变）。

**高级 API (BattleScriptExp)**：
在自定义脚本中，允许调用 `BattleScriptExp` 进行深度干预：
*   `InsertPassiveSkill`：动态被动注入。可在技能释放后，给目标添加临时监听器（如“下次受击时爆炸”）。
*   `SetCustomValue` / `GetCustomValue`：自定义变量存储。用于跨回合或跨技能记录状态。
*   `ModifyAttribute`：在技能执行过程中即时修改英雄属性。

### 3.3 攻击范围与目标选择机制
MiniBattleSimulator 中的“攻击范围”并非具体的距离数值，而是通过**“阵营判定 + 站位筛选 + 目标数量”的规则系统**隐式定义的。

*   **配置位置**：技能 Lua 脚本中的 `targetsSelections` 字段。
*   **阵营枚举 (`castTarget`)**：0(自身), 1(友方全体), 2(敌方全体), 3(除自己外的友方), 4(全场所有人)。
*   **站位限制 (`tSConditions.wpType`)**：
    战场按 `wpType` (1~8) 分为前排(1,2,3)、中排(4,5)、后排(6,7,8)。若配置了特定的 `wpType`（如仅限前排），技能范围将被严格限制。
*   **计算流程**：
    `BattleFormation` 维护存活阵列。默认采用**前排优先**逻辑（若前排死光，范围自动顺延至下一排）。随后根据 `castTarget`、`wpType` 和 `Num`（最大目标数），结合 `tSFilter`（如血量最低等属性过滤）计算出最终的受击目标列表，交由 `SkillExecutor` 结算。

## 4. Buff 系统机制

Buff 系统用于处理战斗中的各种持续性状态、控制效果和属性增减。

### 4.1 静态模板 (`res_buff_template.json`)
*   `buffId`: 唯一标识。
*   `mainType` / `subType`: 区分增益、减益、控制状态（如 `STUN`眩晕、`Frozen`冰冻、`SILENT`沉默）。处于硬控状态的单位在回合判定中将直接跳过行动。
*   `duration`: 默认持续回合。

### 4.2 动态逻辑 (`buff_{ID}.lua`)
#### 触发时机 (`timing` 枚举)
决定了 Buff 效果何时被激活结算：
*   `1 (ON_ADD)` / `2 (ON_REMOVE)`: 获得或移除瞬间触发。
*   `3 (ON_ROUND_BEGIN)`: 回合开始时触发（常用于 **HOT** 持续治疗或回能）。
*   `4 (ON_ROUND_END)`: 回合结束时触发（常用于 **DOT** 毒/流血持续伤害）。
*   `5 (ON_ATTACK)` / `6 (ON_DEFEND)`: 发生攻击/受击行为时触发。
*   `7 (ON_DAMAGE)` / `8 (ON_RECEIVE_DAMAGE)`: 造成或受到伤害时触发（常用于反伤或吸血）。
*   `11 (ON_KILL)` / `12 (ON_DEATH)`: 击杀目标或自身死亡时触发。

#### 效果类型 (`effects`)
定义具体的修改行为，支持以下主流类型：
*   `damage` (DOT): 持续伤害。参数：`value` (伤害值), `damageType` (物理/法术)。
*   `heal` (HOT): 持续治疗。参数：`value` (恢复量，可结合 `percent` 恢复最大生命值百分比)。
*   `attr_change` (ADD_ATTR): 属性变更。参数：`attr` (属性 ID), `value` (正负值)。
*   `energy`: 能量变更。参数：`value` (增减能量)。
*   `dispel`: 驱散。参数：`targetType` (驱散增益或减益)。
*   `custom`: 自定义回调。参数：`func` (脚本内定义的函数名)。

## 5. 核心战斗枚举完整列表

在编写自定义脚本或配置表时，经常需要引用底层的枚举值。以下是系统中完整的枚举定义映射。

### 5.1 目标选择枚举 (`E_CAST_TARGET`)
决定技能释放的阵营倾向（位于 `core/battle_enum.lua`）：
*   `0`: NA (无/无效目标)
*   `1`: Enemy (敌方全体)
*   `2`: Self (自身)
*   `3`: Alias (友方全体)
*   `4`: AlliesExcludeSelf (除自己外的友方)
*   `5`: AliasIncludeCollect (友方含召唤物)
*   `7`: EveryOne (全场所有人)
*   `8`: EveryOneExcludeSelf (全场所有人除自己)

### 5.2 Buff 触发时机 (`E_BUFF_TIMING`)
决定 Buff 效果在什么时间点进行结算（位于 `modules/battle_buff.lua`）：
*   `1`: ON_ADD (获得瞬间触发)
*   `2`: ON_REMOVE (移除/消失瞬间触发)
*   `3`: ON_ROUND_BEGIN (回合开始时触发，适合 HOT)
*   `4`: ON_ROUND_END (回合结束时触发，适合 DOT)
*   `5`: ON_ATTACK (英雄发起攻击时触发)
*   `6`: ON_DEFEND (英雄被攻击时触发)
*   `7`: ON_DAMAGE (造成伤害时触发)
*   `8`: ON_RECEIVE_DAMAGE (受到伤害时触发)
*   `9`: ON_HEAL (产生治疗时触发)
*   `10`: ON_RECEIVE_HEAL (受到治疗时触发)
*   `11`: ON_KILL (击杀目标时触发)
*   `12`: ON_DEATH (自身死亡时触发)

### 5.3 被动技能触发时机 (`E_TRIGGER_TIMING`)
被动技能比普通 Buff 具有更细粒度的战斗流程介入点（位于 `modules/battle_passive_skill.lua`）：
*   `1`: BATTLE_BEGIN (整场战斗开局时)
*   `2`: BATTLE_END (战斗结束时)
*   `3`: ROUND_BEGIN (大回合开始时)
*   `4`: ROUND_END (大回合结束时)
*   `5`: ACTION_BEGIN (轮到该英雄行动前)
*   `6`: ACTION_END (该英雄行动结束后)
*   `7`: HP_CHANGE (血量发生变化时)
*   `20`: ON_SKILL_CAST (释放任何技能时)
*   *(其他 10~19 的枚举与 Buff Timing 类似)*

### 5.4 核心属性 ID (`E_ATTR` / `ATTR_ID`)
用于 `BattleScriptExp.ModifyAttribute` 等函数中操作具体属性（位于 `modules/battle_attribute.lua`）：
*   `1`: HP (当前生命值)
*   `2`: ATK (攻击力)
*   `3`: DEF (防御力)
*   `4`: SPEED (速度/行动力)
*   `5`: CRIT_RATE (暴击率)
*   `6`: CRIT_DMG (暴击伤害)
*   `7`: HIT_RATE (命中率)
*   `8`: DODGE_RATE (闪避率)
*   `9`: DMG_REDUCE (最终伤害减免)
*   `10`: DMG_INCREASE (最终伤害加成)

### 5.5 其他基础枚举
*   **伤害类型 (`E_DMG_TYPE`)**: `1` (普通伤害), `2` (持续伤害 DOT), `3` (附加伤害), `4` (生命流失/扣血不致死), `5` (纯属性真实伤害)。
*   **Buff 主类型 (`E_BUFF_MAIN_TYPE`)**: `1` (增益 GOOD), `2` (减益 BAD), `3` (控制 CONTROL，会跳过回合)。
