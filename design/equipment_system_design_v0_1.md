# 装备系统策划案 v0.1

## 1. 文档信息

- 文档名称：装备系统策划案 v0.1
- 适用范围：Roguelike Run 系统首版装备玩法
- 文档状态：草案 / 可进入实现拆解
- 对齐目标：统一装备系统的设计定位、规则边界、内容结构、数值边界与交互表达

## 2. 文档定位

本文档用于定义当前版本 Roguelike 装备系统的正式设计口径。

当前版本中，装备系统明确采用：

- `Run 级全局被动装备池`
- 非逐角色穿戴
- 非槽位管理系统
- 非词条洗练系统

即：

- 装备一旦获得，即对当前 Run 生效
- 装备不需要穿脱
- 装备不需要指定归属角色
- 装备通过职业过滤对符合条件的队伍单位生效

本文档优先级高于旧文档中与本阶段装备形态冲突的描述。若旧稿中存在“角色装备槽 / 逐角色调整装备归属”等表述，以本文档为准。

## 3. 设计目标

装备系统首版需要解决以下问题：

- 让玩家在一局内形成明确的构筑强化感
- 让职业卡、装备、祝福三类成长资源职责清晰
- 让路线选择与资源获取形成真实取舍
- 让装备强化建立在 5e 核心数值边界内，不破坏命中、AC、Spell DC 的稳定性
- 让系统复杂度保持可控，优先验证玩法闭环而非堆叠复杂机制

首版体验目标：

- 玩家知道自己拿到了什么装备
- 玩家知道这件装备强化了哪些职业
- 玩家知道这件装备为什么值得走这条路线或花这笔钱
- 玩家能在 Boss 前形成至少 1 条清晰的装备强化主题

## 4. 非目标

本阶段不进入以下设计：

- 逐角色装备栏
- 武器 / 防具 / 饰品穿脱管理
- 装备耐久
- 装备强化、升星、洗词条
- 套装系统
- 装备技能树
- 背包容量管理

如未来需要升级为角色槽位装备系统，应另立专项设计，不与本阶段版本混做。

## 5. 系统职责与分工

### 5.1 装备系统职责

装备系统负责：

- 放大已有 Build 优势
- 补足队伍结构短板
- 在局内提供比职业卡更轻量、更频繁的强化反馈
- 为商店、事件、战斗节点提供有价值的横向奖励

### 5.2 与职业卡的分工

职业卡负责：

- 获取新职业单位
- 职业单位进阶
- 职业主成长线推进

装备负责：

- Run 内横向强化
- 围绕职业 archetype 做定向增益
- 提供数值与少量机制型补正

### 5.3 与祝福的分工

祝福负责：

- 更泛用、更抽象的策略强化
- 偏局内规则、节奏、收益修正

装备负责：

- 更具体、更可感知的职业向数值补强
- 更贴近“这一局拿到了一件关键强化”的反馈

## 6. 核心定义

### 6.1 系统形态

装备系统采用“持有即生效”的全局被动模型。

运行时仅需记录：

- 当前 Run 已持有的装备列表

不记录：

- 装备归属角色
- 装备槽位状态
- 穿脱行为
- 装备替换关系

### 6.2 生效方式

每件装备通过以下信息定义其生效范围：

- `rarity`：稀有度
- `slot`：视觉/语义分类，不代表真实装备栏
- `effectType`：效果类型
- `classIds`：适用职业
- `params`：数值参数

生效规则：

- 获得装备后，立即进入当前 Run 的装备池
- 战斗开始前，系统将装备效果统一桥接到战斗单位属性
- 只对符合 `classIds` 的职业生效
- 通用装备可对所有职业生效

### 6.3 重复规则

首版中不允许“纯无效重复装备”。

若玩家获得已持有装备，应转化为补偿收益，而不是静默吞掉。

补偿形式建议为：

- 金币
- 免费刷新
- 少量局内资源

首版默认使用金币补偿。

## 7. 设计原则

### 7.1 低复杂度优先

装备系统应优先验证以下闭环：

- 获取闭环
- 生效闭环
- 展示闭环
- 经济闭环

在这些闭环稳定前，不引入更复杂的装备管理玩法。

### 7.2 强化 Build，而不是取代 Build

装备不能替代职业成长本身。

具体体现为：

- 装备不能直接提供职业卡等价价值
- 装备不能独立撑起完整流派
- 装备应该放大已有职业队伍优势，或修正明显短板

### 7.3 数值边界稳定

装备只对当前已接入的 5e 主轴做微调：

- 命中
- AC
- Spell DC
- 豁免
- 武器额外伤害

避免大倍率叠乘、避免脱离 5e 基线的暴涨。

### 7.4 结果清晰可读

玩家必须能快速理解：

- 装备名称
- 稀有度
- 适用职业
- 核心效果
- 是否为重复转化

## 8. 内容结构

### 8.1 稀有度分层

首版保留三档稀有度：

- `common`
- `rare`
- `boss`

职责划分如下：

| 稀有度 | 职责 | 体验定位 |
| --- | --- | --- |
| `common` | 基础补强 | 稳定、有用、低波动 |
| `rare` | 构筑成型 | 有方向感、能拉开差异 |
| `boss` | 终局确认 | 有记忆点、形成章节签名强化 |

### 8.2 装备类别

首版按语义分为以下类别：

- 武器系
- 防具系
- 盾牌系
- 法器系
- 饰品系

说明：

- `slot` 仅用于内容分类、命名和 UI 展示
- `slot` 不构成真实的角色装备栏约束

### 8.3 装备功能轴

当前版本围绕以下功能轴设计装备：

- 近战输出轴：`hit + 武器伤害附加`
- 远程输出轴：`hit + 武器伤害附加`
- 前排承伤轴：`ac`
- 奥术施法轴：`spellDC`
- 神术支援轴：`spellDC`
- 通用生存轴：`ac + save`

## 9. 首版内容池规划

### 9.1 数量目标

当前正式装备池规模：

- 8 件装备

当前分布：

- 2 件 `common`
- 4 件 `rare`
- 2 件 `boss`

### 9.2 职业覆盖目标

当前正式装备池覆盖以下 archetype：

- 前排物理
- 后排远程
- 奥术施法
- 神术支援
- 通用生存

### 9.3 当前正式装备说明

文档不维护具体装备清单、职业分配和数值表，正式定义以配置文件为准。

当前配置入口：

- `config/roguelike/run_equipment_config.lua`

文档仅保留一个示例：

- 示例：`防护披风`，当前语义为 `AC +1`，`豁免 +1`

## 10. 产出与投放设计

### 10.1 来源总览

装备统一来源为：

- 普通战掉落
- 精英战掉落
- Boss 奖励
- 商店购买
- 事件获得

### 10.2 普通战掉落

普通战承担基础补强职责。

设计原则：

- 有概率掉落
- 以 `common` 为主
- 小概率出现 `rare`
- 不掉 `boss`

建议目标：

- 总掉率维持在 `30% - 40%`

### 10.3 精英战掉落

精英战承担构筑成型职责。

设计原则：

- 必定掉落装备
- 以 `rare` 为主
- 有概率出现 `boss`

精英战应让玩家在高压路线中获得更明确的装备收益预期。

### 10.4 Boss 奖励

Boss 奖励不建议继续走普通池概率逻辑。

建议：

- 直接掉落 `boss` 装备
- 或掉落章节签名件

其职责不是“再给一件装备”，而是“明确这一章的终局强化记忆点”。

### 10.5 商店购买

商店承担定向修正构筑的职责。

设计原则：

- 玩家可以用金币换取更可控的强化
- 商店应承担“补短板”与“确认路线”的功能
- 不允许商店购买体验因为重复装备而变成无效消费

### 10.6 事件获得

事件承担高风险高收益装备投放。

设计原则：

- 用金币、生命、路线风险换取特定装备
- 让事件成为构筑分岔的主动入口
- 避免将事件装备写成纯文案奖励

## 11. 路线节奏目标

以 Act1 为基准，建议装备获取节奏如下：

### 11.1 安全路线

- 全章目标获得 `1 - 2` 件有效装备
- 以稳定通关为主
- 更多依赖恢复、招募、基础经济

### 11.2 战斗路线

- 全章目标获得 `3 - 4` 件有效装备
- 更容易形成明确强化主题
- 压力更高，但回报也更清晰

### 11.3 Boss 前状态

Boss 前理想状态：

- 玩家至少形成 1 条明确装备强化方向
- 玩家能说明当前 Build 的主强化点

## 12. 数值设计规范

### 12.1 平面值边界

单件装备建议边界：

- `hit` 不超过 `+1`
- `ac` 不超过 `+1`
- `spellDC` 不超过 `+1`
- `save` 不超过 `+1`

Act1 局内同轴累计建议边界：

- 不超过 `+2`

### 12.2 额外伤害边界

单件装备建议边界：

- 武器额外伤害：`+1`

Act1 局内同轴累计建议边界：

- 同一单位来自装备的固定额外伤害不超过 `+2`

### 12.3 防护边界

单件装备建议：

- `AC +1`

### 12.4 设计原则

数值设计遵循以下原则：

- 优先做平面微调
- 优先使用当前运行时已接入的固定值 5e 语义
- 避免装备单独造成数值碾压
- 任何一件装备都应能被玩家理解其收益来源

## 13. 经济设计

### 13.1 商店价格建议

首版建议价格区间：

| 稀有度 | 建议价格 |
| --- | --- |
| `common` | `60 - 75` 金 |
| `rare` | `85 - 100` 金 |
| `boss` | `120 - 140` 金 |

### 13.2 重复补偿建议

重复装备默认转化为金币：

| 稀有度 | 建议补偿 |
| --- | --- |
| `common` | `30` 金 |
| `rare` | `50` 金 |
| `boss` | `80` 金 |

设计要求：

- 补偿必须可见
- 补偿必须在获得瞬间明确提示
- 玩家不能在事后推测“刚才到底有没有拿到收益”

### 13.3 事件代价建议

若事件直接给予装备，建议代价边界：

- 金币：`50 - 70`
- 生命：`15% - 20%`

一般不建议超过：

- `25%` 当前生命代价

## 14. 交互与 UI 设计要求

### 14.1 信息页

信息页新增“已拥有装备”区域。

每件装备至少展示：

- 名称
- 稀有度
- 适用职业
- 效果摘要
- 是否重复转化

### 14.2 商店页

商店商品卡片至少展示：

- 名称
- 稀有度
- 价格
- 适用职业
- 核心效果摘要
- 是否已拥有
- 若重复购买将转化为什么收益

### 14.3 战斗结算

战斗结算不应只显示“掉落装备数量”。

应直接展示：

- 实际获得的装备名
- 稀有度
- 若为重复，则显示补偿结果

示例：

- 获得：`+1 长剑`
- 重复装备：`防护披风`，已转化为 `50 金币`

### 14.4 队伍页辅助表达

队伍页可选增强表达：

- 在角色卡上显示“受哪些全局装备影响”的标签

其目的不是增加信息负担，而是帮助玩家建立：

- 装备强化对象认知
- Build 强化路径认知

## 15. 数据与配置要求

### 15.1 配置字段

首版装备配置至少包含：

- `id`
- `code`
- `name`
- `rarity`
- `slot`
- `effectType`
- `params`

策划语义层的 `params` 至少支持：

- `classIds`
- `hitDelta`
- `acDelta`
- `spellDCDelta`
- `saveDelta`
- `weaponDamageBonus`
- `spellDamageBonus`
- `healingBonus`
- `damageResistance`
- `reactionDamageReduction`

说明：

- 若实现层暂时保留旧字段名，仅可视为兼容映射
- 策划文档、UI 文案、数值讨论统一使用 5e 语义字段

### 15.2 快照字段

前端快照建议至少补充以下可读字段：

- `effectSummary`
- `applicableClasses`
- `isDuplicate`
- `duplicateCompensation`
- `source`

目标是让前端无需二次推导复杂逻辑，也能清晰展示装备信息。

## 16. 算法与规则要求

### 16.1 稀有度抽取规则

首版必须保证稀有度抽取逻辑准确。

要求：

- 先确定目标稀有度
- 再从对应稀有度池中抽取

不应使用“按最大 tier 从累计池抽取”的方式，否则高稀有装备的真实命中率会被低稀有内容稀释。

### 16.2 重复装备处理规则

所有入口统一遵循重复处理规则：

- 战斗掉落
- 商店购买
- 事件获得
- 奖励池选择

统一要求：

- 检测是否已拥有
- 若重复，立即转化为补偿
- 将补偿结果写入结算提示

### 16.3 Boss 装备投放规则

Boss 奖励应从 `boss` 专属池中独立结算，避免与普通池混用。

## 17. 验收标准

装备系统首版验收至少满足以下标准：

### 17.1 规则验收

- 装备系统按 `Run 级全局被动装备池` 运行
- 不存在逐角色穿脱依赖
- 所有装备获取入口遵循统一重复处理规则

### 17.2 UI 验收

- 玩家可在信息页查看完整已拥有装备列表
- 玩家可在商店页理解装备效果与适用职业
- 玩家可在战斗结算中看到实际获得结果，而不是只有数量

### 17.3 数值验收

- Act1 中装备不会导致明显碾压
- `hit / ac / spellDC / save` 累计仍在可控边界内
- 装备强化感明显，但不会取代职业成长主轴

### 17.4 玩法验收

- 一章内商店、事件、战斗三种来源都能实际验证装备投放
- 玩家能形成至少 1 条明确装备强化主题
- 路线选择能体现“低风险稳健”和“高风险高回报”的差异

## 18. 分期建议

### 18.1 Phase 0：规则修补

目标：

- 修正稀有度抽取
- 补齐重复装备补偿
- 补充战斗结算展示

### 18.2 Phase 1：MVP 成型

目标：

- 稳定当前 8 件正式装备的获取、展示与重复补偿闭环
- 信息页与商店页补齐装备表达
- 地图补足商店 / 事件实际投放入口

### 18.3 Phase 2：内容强化

目标：

- 增加章节签名装备
- 增加高风险高收益事件装备
- 增强路线差异化投放

### 18.4 Phase 3：后续增强

目标：

- 增加少量机制型装备
- 做装备与祝福的轻联动

本阶段仍不进入角色穿戴、套装、词条养成系统。

## 19. 风险与待同步项

### 19.1 当前主要风险

- 当前 UI 若只显示数量，会削弱装备系统存在感
- 若不处理重复装备，将明显伤害商店和掉落体验
- 若后续扩池未继续遵循当前 5e 语义字段，口径会再次发散

### 19.2 已同步文档

本次已同步以下文档，保证口径一致：

- `roguelike_run_system_design.md`
- `class_system_design.md`
- `class_promotion_design.md`
- `implementation_guidelines.md`

## 20. 结论

装备系统首版应明确走：

- `Run 级全局被动装备池`

先完成：

- 规则收口
- UI 补齐
- 当前 8 件正式装备的展示闭环
- 三来源投放落地

在此基础上，再评估是否需要进入更高复杂度的装备管理玩法。
