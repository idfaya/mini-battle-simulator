# Roguelike 玩法设计文档

## 1. 文档目的

本文档定义当前版本 Roguelike 玩法的正式产品规则，用于统一策划、程序、测试和 Web 表现的理解。

本文档回答四个问题：

1. 玩家在 Run 中到底在玩什么
2. 每一类节点承担什么职责
3. 成长、恢复、扩编、资源分别通过什么系统获得
4. 第一章当前的目标体验与平衡方向是什么

本文档是玩法设计文档，不展开底层代码实现细节。
程序落地请参考 [roguelike_convergence_program_design.md](file:///c:/work/MiniBattleSimulator/design/roguelike_convergence_program_design.md)。

## 2. 玩法定位

当前 Roguelike 的核心定位不是“奖励池驱动”，而是：

- 地图节点推进
- 小队资源管理
- 战后定向构筑成长
- 路线分化带来的风险与收益选择

玩家每次 Run 的主要决策不再是“打完拿什么杂项奖励”，而是：

- 这一层走哪条路线
- 要不要提前补人
- 要不要在商店/事件拿装备或祝福
- 每场战后这一张 feat 应该给谁

一句话定义：

`这是一个以战后 feat 三选一为主成长、以路线节点为主资源结构的轻量队伍 Roguelike。`

## 3. 核心循环

单次 Run 的标准循环如下：

1. 在地图上选择下一节点
2. 进入节点并结算节点内容
3. 如果是战斗节点并获胜：
   - 自动执行战后固定小休
   - 进入 feat 三选一
4. 处理完奖励或节点结果后回到地图
5. 重复推进直到 Boss 或团灭

玩家单局体验节奏应为：

- 前期建立基础阵容
- 中期形成 1 到 2 个清晰 build 方向
- 后期依靠装备、祝福和关键 feat 顶住 Boss 压力

## 4. 系统职责拆分

为避免一个奖励入口承担过多职责，当前玩法严格拆分为四条线：

### 4.1 成长

- 战后只给 `feat 三选一`
- 每次只选择 1 个成长卡
- 每张卡本质上是：`指定英雄 + 升 1 级 + 获得 1 个 feat`

### 4.2 恢复

- 每次战斗胜利后自动执行固定小休
- 小休是基础恢复手段，不再占奖励位

### 4.3 扩编

- 招募只来自 `recruit` 节点
- 不再放入战斗奖励
- 不再作为商店主商品

### 4.4 资源强化

- 装备来自商店、事件和后续专用掉落入口
- 祝福来自事件、营地和商店
- 金币用于调度资源，不直接替代成长

## 5. 战斗后规则

### 5.1 战后固定小休

战斗胜利后，系统在 feat 选择前自动执行恢复。

Act 1 当前规则：

- 存活英雄恢复 `50%` 最大生命
- 清普通技能冷却
- 恢复大招次数到上限
- 不复活死亡英雄

设计目的：

- 保证每场战斗结束后玩家都有“重新进入可战状态”的基础能力
- 把单场战斗体验和跨战续航压力分开
- 让玩家把注意力集中在 build 选择，而不是被连续掉血压垮

### 5.2 战后 feat 三选一

战后奖励只保留 feat 卡。

每张卡体现为：

- 一个可升级英雄
- 升到的下一级
- 本次获得的 feat

该系统的目标不是随机惊喜，而是稳定构筑。

设计要求：

- 三张卡必须体现一定分化
- 玩家应能明确知道“这次强化的是谁”
- 关键职业 build 必须能稳定在中前期形成

## 6. 节点设计

### 6.1 普通战斗节点

职责：

- 提供基础推进压力
- 触发战后 feat 成长
- 为整章节奏提供主干

体验要求：

- 会消耗资源，但不应该频繁逼近团灭
- 应该让玩家能看见当前 build 是否顺手

### 6.2 精英战斗节点

职责：

- 提供更高的战斗压力测试
- 放大战后成长的价值
- 让路线差异成立

体验要求：

- 明显比普通战更危险
- 但不能成为“稳定筛人点”
- 失败应来自 build 或操作问题，而不是数值断崖

### 6.3 Boss 节点

职责：

- 检验本局 build 是否成型
- 提供章节完成感

体验要求：

- 允许减员
- 不应要求完美状态才能打
- 玩家在 Boss 前应至少拥有一个明显成型的核心组合

### 6.4 招募节点

职责：

- 扩编
- 补位
- 提供队伍结构调整机会

规则：

- 进入节点后弹出招募三选一
- 招募优先补入上阵空缺
- 若队伍已满则进入候补

设计意义：

- 把“补人”从战斗奖励中抽离
- 让路线价值更清晰
- 让玩家主动为了队伍结构走特定路线

### 6.5 商店节点

职责：

- 用金币换取稳定资源强化

当前主要内容：

- 装备
- 祝福
- 服务型商品

商店不承担：

- 主成长
- 主招募

### 6.6 事件节点

职责：

- 提供可选代价与收益
- 让路线不只有纯数值战斗

事件更适合承载：

- 金币
- 装备
- 祝福
- 分支战斗

### 6.7 营地节点

营地的职责已经收口，不再是“第二份小休”。

当前主定位：

- 复活 1 名英雄
- 该英雄回满血、CD、状态与大招次数
- 提供祝福型强化

设计原则：

- 战后小休负责基础续航
- 营地负责翻盘与纠错

## 7. 成长系统

### 7.1 角色成长

角色成长严格围绕 feat 展开。

成长入口：

- 战后三选一

成长效果：

- 升级
- 获得职业 feat
- 由 feat 驱动属性、技能、被动变化

### 7.2 队伍成长

队伍成长来自三层叠加：

- 单人 feat 成长
- 招募补充角色结构
- 装备 / 祝福提供全队或职业向加成

### 7.3 装备系统

当前版本中，装备是 `run 级全局被动资源`，不是逐角色装备栏系统。

这意味着：

- 装备一旦获得，即作用于整个 Run
- 不需要穿脱
- 不需要武器/防具/饰品槽位

当前这样设计是为了控制复杂度。
未来如需扩展为角色槽位装备，应另立专项设计，不在本阶段内。

## 8. 资源系统

### 8.1 金币

金币用于：

- 商店购买装备
- 商店购买祝福
- 支付事件代价

金币不应直接替代 feat 成长价值。

### 8.2 生命与死亡

- 存活英雄会在战后自动恢复一部分生命
- 死亡英雄不会被战后小休复活
- 营地和少数服务承担复活功能

### 8.3 候补

- 招募所得角色可能进入候补
- 候补用于后续替换或补位
- 候补是队伍冗余，不是主战资源

## 9. 路线设计

### 9.1 Safe 路线

定位：

- 压力较低
- 恢复与事件更多
- 更适合稳健通关

目标体验：

- 通关率高
- build 上限略低
- 容错高

### 9.2 Combat 路线

定位：

- 战斗更密集
- 精英压力更高
- 更强调队伍强度与 build 完整度

目标体验：

- 明显更难
- 但必须可通
- 风险换来的应是更快的成长兑现与更高的资源收益空间

## 10. 第一章目标体验

第一章是教学章，不是筛选章。

第一章需要完成的体验任务：

- 教会玩家地图推进
- 教会玩家战后 feat 三选一的价值
- 教会玩家招募、商店、营地分别负责什么
- 让玩家第一次感受到 build 形成

当前平衡目标：

- `safe` 路线：稳定通关
- `combat` 路线：更难但可通，不再稳定死在 `Dark Cabal`

当前重点节点：

- `101102 / Dark Cabal`
- `101201 / Frozen Gate`

## 11. 当前已确认规则

以下规则视为当前版本定案：

- 战后只给 feat 三选一
- 每战后自动小休
- 招募只在招募节点
- relic 对外统一改名为 equipment
- 营地主功能为“复活一名英雄并回满血、CD、状态”
- `rewardGroup` 不再作为 battle / elite / boss 的正式掉落主链，仅保留给 event 等非战斗奖励入口使用
- `team_heal_pct` 不再作为战后奖励类型，仅允许在 event / shop 等独立系统中出现

## 12. 暂不进入本轮范围

以下内容明确不在本轮玩法设计交付范围内：

- 角色装备槽系统
- rewardGroup 作为正式掉落主链的完整恢复
- 跨章节 meta 成长
- 更复杂的事件树与章节随机化
- 多章节不同地图机制

## 13. 后续扩展方向

后续迭代建议按以下顺序推进：

1. 补齐正式装备掉落入口
2. 补齐高级装备效果运行时
3. 增强 feat 卡面的结果可读性
4. 扩展更多招募节点和招募池
5. 为第二章建立不同于第一章的地图与压力结构

## 14. 验收标准

这套玩法设计成立的标志是：

- 玩家能清楚分辨成长、恢复、扩编、资源的来源
- 玩家不会再把战后奖励理解为杂项奖励池
- 第一章 `combat` 路线不再是断崖失败
- 营地节点拥有清晰且不可替代的存在价值
- 招募节点成为真正的队伍构建决策点
