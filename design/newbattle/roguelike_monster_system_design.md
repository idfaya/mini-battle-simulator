# MiniBattle Roguelike 怪物系统策划案

## 1. 文档范围

- 本文档定义 `MiniBattle` 当前 Roguelike 怪物系统的结构、职责边界与后续扩展方向。
- 本文档是以下文档的补充专题：
  - `roguelike_random_generation_design.md`
  - `roguelike_random_battle_parameter_table.md`
  - `roguelike_run_system_design.md`
- 本文档聚焦 `敌人本体`、`怪物编组`、`战斗难度语义` 三个层面，不展开玩家职业成长系统。

---

## 2. 设计目标

### 2.1 当前目标

- 明确当前怪物系统已经具备哪些能力。
- 统一“职业 / 等级 / CR / MonsterType / role”几个概念的职责边界。
- 明确当前随机地图版本下，怪物系统的瓶颈在哪里。
- 为 Act1 后续补怪、扩池、调平衡提供统一口径。

### 2.2 长期目标

- 建立一套符合 5e 规则表达的敌人系统。
- 让敌人不只是“数值块”，而是具备清晰职责和章节生态。
- 让随机地图、随机战斗和随机怪物编组能长期扩充，而不是每补一个怪都要重写关卡逻辑。
- 让怪物系统同时支持：
  - 固定图关卡
  - Roguelike 随机图
  - 精英/Boss 专项设计

---

## 3. 当前系统总览

当前 Roguelike 怪物系统可拆为四层：

```text
原始敌人数据层
→ res_enemy.json / EnemyData

敌人数值构建层
→ Class + Level + MonsterType + 5e 规则

Roguelike 怪物编组层
→ enemy_pick_pool + wave_group_template + formation_profile

战斗实例层
→ battle_template + encounter_pool + runtime enemy group
```

### 3.1 原始敌人数据层

- 敌人基础数据来自 `res_enemy.json`。
- 每个敌人至少包含：
  - `ID`
  - `EnemyName`
  - `Class`
  - `Level`
  - `MonsterType`
  - `SkillIDs`
- 这一层负责“怪是谁”，不负责决定本局会不会遇到它。

### 3.2 敌人数值构建层

- 这一层负责把原始敌人转换成战斗单位。
- 核心变量：
  - `Class`
  - `Level`
  - `MonsterType`
  - 5e 属性模板
- 输出：
  - HP
  - ATK / DEF
  - AC / hit / spell DC
  - 三系豁免
  - 技能列表
  - 相关战斗附加属性

### 3.3 Roguelike 怪物编组层

- 这一层负责回答：
  - 这场战斗具体上哪些怪？
  - 前后排怎么排？
  - 哪一波出现 Boss？
  - 护卫和增援怎么补？
- 核心结构：
  - `enemy_pick_pool`
  - `wave_group_pool`
  - `wave_group_template`
  - `formation_profile`

### 3.4 战斗实例层

- 这一层把战斗模板和怪物编组整合成一场真实战斗。
- 输出：
  - `battle_template`
  - `wave_group_ids`
  - `encounter`
  - `runtime enemy group`

## 4. 核心概念定义

### 4.1 Class

- `Class` 是敌人的底层职业模板来源。
- 它决定敌人使用哪组 5e 风格成长规则。
- `Class` 影响：
  - 护甲计算
  - 命中成长
  - 法术 DC
  - 豁免熟练
  - hit die
  - 基础数值模板

设计结论：

- 敌人“有职业”，但当前职业主要是数值模板语义。
- 敌人职业不等于玩家职业系统中的完整外循环单位。

### 4.2 Level

- `Level` 是敌人战斗构建等级。
- `Level` 会真实影响：
  - HP
  - 熟练加值
  - 命中
  - AC
  - spell DC
  - 豁免
  - 技能解锁段

设计结论：

- 敌人“有等级”，而且等级是真正参与战斗数值构建的。
- 敌人等级不是 Run 中成长出来的，而是战斗生成时指定的模板等级。

### 4.3 MonsterType

- `MonsterType` 表示敌人的大类型分层。
- 当前定义：
  - `0 = Normal`
  - `1 = Elite`
  - `2 = Boss`
- 它负责在同一职业模板基础上施加轻量差异。

设计结论：

- `MonsterType` 是“类型修正层”，不是完整职业层。
- 它应始终保持轻量，不应承担主要难度设计工作。

### 4.4 CR

- `CR` 当前已经存在，并与 `XP` 一起作为元数据维护。
- 它用于表达该敌人的大致危险度。

设计结论：

- 当前 `CR` 主要是标注信息与设计参考。
- 当前随机怪物系统并未严格按照 CR 预算自动拼怪。
- 现阶段压强主控仍是 `budget.difficulty + pressureFactor`。

### 4.5 role

- `role` 表示战斗职责语义。
- 当前示例：
  - `fodder`
  - `skirmisher`
  - `brute`
  - `frontliner`
  - `caster`
  - `elite_caster`

设计结论：

- `role` 当前是“说明性标签”，不是强规则约束。
- 后续应将其升级为正式的编组约束标签体系。

## 5. 当前怪物系统已具备的能力

### 5.1 5e 数值化能力

- 敌人已经不是纯静态面板，而是按 5e 逻辑构建。
- 已具备：
  - 能力值
  - 熟练加值
  - 命中 / AC / spell DC
  - hit die 驱动 HP
  - 豁免成长

### 5.2 技能成长能力

- 敌人支持按等级解锁同技能家族的更高段技能。
- 精英与 Boss 还能更早解锁更强技能层级。

### 5.3 Roguelike 随机编组能力

- 当前已支持：
  - 波次组池
  - 波次模板
  - 前后排槽位
  - Boss 最终波强制出现
  - 护卫补位
  - 单怪重复数限制

### 5.4 固定图与随机图兼容能力

- 固定图仍可继续使用固定 `battle_id -> encounter` 链路。
- 随机图已支持 `battle_pool -> battle_template -> wave_group -> runtime group`。

## 6. 当前内容现状

### 6.1 当前怪物总量

- 当前基础怪物池为 `7` 个：
  - Slime
  - Goblin
  - Orc
  - Skeleton
  - DarkMage
  - IceDemon
  - ThunderLord

### 6.2 当前内容分布

- 普通怪偏少。
- 远程/法系怪偏少。
- 明确功能怪不足。
- 精英怪数量不足。
- Boss 层缺少横向变化。

### 6.3 当前章节生态特征

- 目前 Act1 的怪物生态更像“少量基础怪 + 少量法系高压怪”。
- 玩家可以感知到战斗系统变化，但难以感知章节怪物生态的层次变化。

---

## 7. 当前系统的主要问题

### 7.1 内容量不足

- 现在的怪物数量足以验证系统，但不足以支撑长期随机体验。
- 多跑几局后，玩家会频繁看到相同怪反复换排列组合。

### 7.2 职责标签没有真正进入约束系统

- 当前编组仍主要依赖“池子命名”表达前后排和职责。
- 没有真正基于 `role/tag` 做严格校验。

表现为：

- 前排池、后排池语义清晰，但约束能力有限。
- 后续补怪越多，越容易出现“不合理组合混入池子”的问题。

### 7.3 CR 与实际预算没有闭环

- 当前 CR 是存在的，但没有成为编组预算的硬约束。
- 随机怪物系统的主难度控制仍依赖 encounter budget。

风险：

- 新补怪物时，CR 与真实压强可能逐步脱节。

### 7.4 MonsterType 承担了过多“强度表达”

- 当前普通 / 精英 / Boss 的差异，部分依赖 `MonsterType` 修正。
- 如果后续继续把难度差异都压到这一层，会弱化怪物本体设计。

### 7.5 Boss 生态偏薄

- 当前 Boss 候选池过窄。
- 护卫池与 Boss 本体之间缺乏更明确的语义配套关系。

---

## 8. 设计原则

### 8.1 怪物系统不发明新数学

- 继续服从现有 5e 规则实现。
- 继续以 `budget.difficulty` 与 `budget.pressureFactor` 作为整场压强主控。
- 不通过叠加神秘倍率来“修怪物”。

### 8.2 怪物设计与编组设计分层

- 单个怪物负责自身身份和玩法。
- 编组模板负责“这场战斗想表达什么压力”。
- Encounter 负责整体难度带。

### 8.3 Boss 必须保持强识别度

- Boss 不应只是“大号普通怪”。
- Boss 必须在以下至少两项上和普通怪拉开差异：
  - 技能结构
  - 波次位置
  - 护卫关系
  - 战斗规则

### 8.4 章节怪物应形成生态

- 每章应形成若干可辨识的敌方族群。
- 怪物不是独立散点，而应服务于章节主题。

## 9. Act1 平衡调优经验（2026-05）

### 9.1 数量感与强度必须拆开处理

- 本轮 Act1 调优最重要的结论是：`怪物数量` 和 `战斗强度` 不能绑死在一起。
- 在 Roguelike 战斗里，数量首先承担的是：
  - 波次氛围
  - 阵线压迫感
  - “这是一场遭遇”而不是“几只怪排队送掉”的观感
- 真正决定能不能打过的主控变量，应优先放在：
  - 敌方等级
  - 遭遇预算
  - 护卫质量
  - Boss 阶段压强

设计结论：

- 如果目标是“怪多但不离谱”，优先降 `Level`，不要先砍单位数。
- 如果目标是“每波至少 4 只”，就必须允许普通战通过等级回调来消化压力。

### 9.2 普通战优先用等级控强度

- 本轮实际调优中，晚期普通战在保留 `3 波`、保留高密度编组时，单纯降低 `pressureFactor` 与轻量倍率不够稳定。
- 真正有效的做法是：
  - 保留波次数与单波数量
  - 把晚期普通战的敌方实际等级下压
  - 让普通战的等级钳制规则允许“低于当前队伍等级更多一点”

设计结论：

- 普通战与事件战应允许比队伍等级更低的敌方等级下限。
- Boss 战可以保留更紧的等级下限，但普通战不应和 Boss 共用过紧的等级钳制。

### 9.3 连续战斗要按“残血队伍”验证

- 本轮失败点并不总是出在“单场超模”，而是经常出在：
  - 前两场普通战后已有减员
  - 路线没有提前招募
  - 两次营地只能做有限恢复
  - 队伍以残血状态进入 L6 / L7 连续战斗
- 这说明 Roguelike 调平衡不能只看单场满状态模拟，必须看：
  - 资源继承
  - 死亡继承
  - Camp 恢复后状态
  - 连续两场战斗后的存活人数

设计结论：

- 普通战模板的验收口径，不是“满编满血是否能过”，而是“典型残血状态下是否仍有通关窗口”。
- L6 / L7 这种连续战斗层，必须按低资源状态做回归。

### 9.4 Boss 池必须和精英池严格隔离

- 本轮调试中出现过 Boss 节点误走精英遭遇的问题。
- 这个错误的后果不是“略微变难”，而是会直接导致：
  - Boss 本体逻辑和遭遇预算脱节
  - 护卫质量异常升高
  - 最终战压力远高于设计预期

设计结论：

- Boss 节点必须只从 Boss 专用 `encounter_pool` 中取值。
- 不能把“精英后段高压遭遇”拿来临时顶 Boss 遭遇。
- `battle_template.kind = boss`、`encounter.kind = boss`、`boss_phase_group` 三者必须形成闭环。

### 9.5 Boss 压强优先削护卫质量，不先削 Boss 本体数量感

- 本轮 Boss 的真实问题不是“Boss 本体太强”这么简单，而是：
  - 三波总量大
  - 尾波护卫质量偏高
  - 容易出现“Boss + 高质量护卫”的叠压
- 这种情况下，如果直接砍 Boss 本体，会破坏终局识别度；更合适的做法是先削：
  - 前两波杂兵质量
  - 尾波护卫质量
  - 护卫池中高压单位的占比

设计结论：

- Boss 调优顺序优先级建议为：
  1. 检查 Boss 是否走对池
  2. 下调护卫质量
  3. 下调 Boss 实际等级
  4. 最后才考虑改 Boss 本体波次结构

### 9.6 调优顺序应固定，避免“到处一起砍”

- 本轮调优里最容易浪费时间的情况，是同时改：
  - 波次数
  - 单波数量
  - 等级
  - 预算
  - 护卫池
  - 恢复曲线
- 这样会让问题来源变得不可判断。

推荐调优顺序：

1. 先确认池子是否走对。
2. 再确认失败发生在普通战、精英战还是 Boss 战。
3. 普通战优先调 `Level`，不先砍数量。
4. Boss 优先调护卫质量，再调 Boss 等级。
5. 只有当战斗本体已合理、但仍然因跨战损耗失败时，再去调恢复曲线。

一句话总结：

- `数量保氛围，等级控强度，Boss 先削护卫不先削本体。`

## 10. 后续目标结构

### 10.1 怪物应分为三层内容池

- 普通怪
  - 用于填充大部分战斗与波次主体
- 精英怪
  - 用于中后段压强节点和高收益战
- Boss 怪
  - 用于章节终局与关键战斗

### 10.2 怪物应覆盖六类职责

- 前排承伤
- 近战输出
- 机动骚扰
- 远程持续输出
- 法系爆发 / 控场
- 功能辅助 / 召唤 / 治疗

### 10.3 编组约束应升级为标签驱动

后续建议把当前 `role` 扩展为更明确的标签：

- `frontliner`
- `brute`
- `skirmisher`
- `ranged`
- `caster`
- `support`
- `summoner`
- `undead`
- `beast`
- `boss`
- `guard`

然后正式支持：

- `required_tags`
- `forbidden_tags`
- `boss_only`
- `elite_only`
- `max_same_enemy`

## 11. Act1 目标规模建议

### 11.1 最低可用规模

若目标只是让随机图“有明显变化感”，Act1 建议至少扩到 `10` 个怪：

- 普通怪：`6~7`
- 精英怪：`2`
- Boss：`1`

### 11.2 推荐规模

若目标是让 Act1 真正具备重复游玩价值，建议扩到 `12~15` 个怪：

- 普通怪：`8~10`
- 精英怪：`3~4`
- Boss：`1~2`

### 11.3 Act1 建议补位方向

优先补以下类型：

- 轻前排怪 1 个
- 远程射手怪 1 个
- 功能型辅助怪 1 个
- 召唤或控场怪 1 个
- 中压精英怪 1 个

设计原因：

- 这些补位能最快提升随机编组的辨识度。
- 比继续补“又一个近战肉怪”更划算。

## 12. Act1 怪物生态建议

Act1 建议拆成三个生态层：

### 12.1 杂兵生态

- 用于早期普通战
- 强调：
  - 数量感
  - 低血线压力
  - 前后排基础教学

### 12.2 中段压强生态

- 用于中段普通战、事件战、第一批精英战
- 强调：
  - 近战与后排联动
  - 法系威胁开始成型
  - 玩家需要开始考虑队伍结构与容错

### 12.3 终局高压生态

- 用于末段精英与 Boss
- 强调：
  - 单体高威胁
  - 护卫与 Boss 联动
  - 最终波识别度

---

## 13. 开发建议顺序

### 13.1 第一阶段

- 不改底层 5e 数值构建。
- 只扩怪物内容量与标签。
- 目标：让现有随机系统更“像随机系统”。

### 13.2 第二阶段

- 将 `role` 升级为正式编组标签。
- 把 `formation_profile` 从“前后排池”升级为“前后排池 + 标签约束”。

### 13.3 第三阶段

- 讨论是否引入 `CR budget` 作为编组辅助约束。
- 注意：CR 只能做辅助，不建议替代 encounter budget。

---

## 14. 结论

- 当前怪物系统在工程结构上已经合格。
- 它已经具备：
  - 职业模板
  - 等级成长
  - MonsterType 分层
  - CR 元数据
  - 技能成长
  - Roguelike 随机编组
- 当前最大的短板不是“系统没有”，而是：
  - 内容量不足
  - 标签约束不足
  - 章节生态不够厚

因此后续怪物系统工作的优先级应为：

1. 扩内容量
2. 升级标签约束
3. 再讨论 CR 预算化闭环

一句话总结：

- 当前怪物系统已经是“能跑的完整框架”；
- 下一阶段要做的是把它从“框架可用”推进到“内容好玩”。
