# MiniBattle Roguelike Run 系统设计文档

## 1. 文档范围

- 本文档定义 `MiniBattle` 的 Roguelike Run 系统。
- 本文档采用 `minibattle_combat_design_document_v_1.md` 作为上位规则源。
- 本文档定义：
  - Run 主循环
  - 章节地图
  - 路线分化
  - 节点类型
  - 战斗结算
  - 职业卡
  - 装备
  - 金币
  - 队伍状态
  - 招募 / 商店 / 事件 / 营地
  - 章节完成与 Run 结算

---

## 2. Run 主循环

单次 Run 统一采用以下流程：

```text
进入章节地图
→ 选择下一节点
→ 进入节点并结算
→ 若为战斗并胜利：
   → 固定恢复
   → 职业卡三选一
→ 返回地图
→ 继续推进
→ 章节 Boss
→ 章节结算
→ Run 结束
```

---

## 3. 章节地图

### 3.1 地图结构

- 每章采用节点地图推进制。
- 地图由路径连接的节点构成。
- 玩家每次从当前节点选择下一步。
- 当前章节存在唯一起点和唯一期末 Boss 节点。

### 3.2 章节节点类型

每章至少包含以下节点类型：

- 普通战
- 精英战
- Boss
- 招募
- 商店
- 事件
- 营地

### 3.3 章节推进规则

- 当前节点结算完成后，解锁下一层可到达节点。
- 玩家到达 Boss 节点并获胜后，当前章节完成。
- 当前队伍全灭后，本次 Run 结束。

### 3.4 章节节点表

每个地图节点对应一条节点记录。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | 节点编号 |
| `act` | string | 所属章节 |
| `type` | enum | `normal_battle` / `elite_battle` / `boss_battle` / `recruit` / `shop` / `event` / `camp` |
| `route` | enum | `safe` / `high_pressure` / `boss_path` |
| `battle_id` | string | 战斗节点绑定的战斗编号，非战斗节点为空 |
| `shop_id` | string | 商店节点编号，非商店节点为空 |
| `event_id` | string | 事件节点编号，非事件节点为空 |
| `camp_id` | string | 营地节点编号，非营地节点为空 |
| `recruit_id` | string | 招募节点编号，非招募节点为空 |
| `reward_gold` | integer | 节点固定金币收益 |
| `reward_equip_count` | integer | 节点装备掉落次数 |
| `next_nodes` | string[] | 可到达的下一节点列表 |

## 4. 路线结构

### 4.1 基础路线

每章至少包含两条主路线：

- `稳健路线`
- `高压路线`

### 4.2 稳健路线

- 普通战占比更高
- 营地与商店节点占比更高
- 招募节点更容易出现
- 章节推进更稳定

### 4.3 高压路线

- 精英战占比更高
- 事件与高收益节点占比更高
- 战损压力更高
- 成型速度与上限更高

### 4.4 路线切换

- 同一章内允许在分支处切换路线
- 路线选择只影响后续可达节点

## 5. 节点系统

### 5.1 普通战节点

- 进入一场普通战斗
- 通过 `battle_id` 进入战斗主表
- 胜利后执行固定恢复
- 胜利后进入职业卡三选一

### 5.2 精英战节点

- 进入一场精英战斗
- 通过 `battle_id` 进入战斗主表
- 胜利后执行固定恢复
- 胜利后进入职业卡三选一
- 节点金币与装备收益高于普通战

### 5.3 Boss 节点

- 进入当前章节 Boss 战
- 通过 `battle_id` 进入战斗主表
- 获胜后完成当前章节
- 章节结束后进入章节结算

### 5.4 招募节点

- 进入招募结算
- 提供职业单位补充入口
- 优先承担补位与扩编职责
- 具体配置采用 `roguelike_service_parameter_table.md`

### 5.5 商店节点

- 进入商店结算
- 使用金币购买装备与服务
- 具体配置采用 `roguelike_service_parameter_table.md`

### 5.6 事件节点

- 进入事件结算
- 在多个结果中选择一项
- 结果可包含金币、装备、招募、战斗或状态变化
- 具体配置采用 `roguelike_service_parameter_table.md`

### 5.7 营地节点

- 进入营地结算
- 执行复活、深度恢复或整队修复
- 具体配置采用 `roguelike_service_parameter_table.md`

---

## 6. 战斗节点结算

### 6.1 战斗胜利结算顺序

```text
战斗胜利
→ 发放节点金币
→ 结算节点掉落
→ 固定恢复
→ 职业卡三选一
→ 返回地图
```

### 6.2 节点与战斗表接口

战斗节点统一通过以下关系接入单场战斗表：

```text
node.id
→ node.battle_id
→ single_battle_parameter_table.id
```

### 6.3 固定恢复

战斗胜利后统一执行：

- 存活单位恢复一定比例生命
- 清普通技能冷却
- 保留死亡状态

### 6.4 节点掉落

战斗节点可提供：

- 金币
- 装备掉落

职业卡不进入随机掉落池，统一由战后结算发放。

### 6.5 战斗结算输出

战斗节点结算后，统一输出以下结果给 Run 层：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `battle_result` | enum | `win` / `lose` |
| `dead_units` | string[] | 本场死亡单位 |
| `survive_units` | string[] | 本场存活单位 |
| `hp_snapshot` | table | 本场结束生命快照 |
| `reward_gold` | integer | 本节点金币收益 |
| `reward_equip_count` | integer | 本节点装备掉落次数 |
| `enter_card_reward` | boolean | 是否进入职业卡三选一 |

---

## 7. 职业卡系统

### 7.1 职业卡入口

- 所有战斗胜利后进入职业卡三选一
- 玩家每次只选择一张职业卡
- 职业卡是 Run 内职业成长主入口

### 7.2 职业卡结算结果

职业卡只会结算为两类结果：

- 新职业单位
- 同职业进阶

### 7.3 新职业单位

- 当前未持有该职业时，结算为新职业单位
- 队伍未满时直接上阵
- 队伍已满时进入候补

### 7.4 同职业进阶

- 当前已持有该职业时，结算为同职业进阶
- 进阶顺序统一为：

```text
低阶
→ 中阶
→ 高阶
```

- 当前已达高阶的职业不再进入职业卡池

### 7.5 职业卡显示信息

每张职业卡至少显示：

- 职业头像
- 职业名称
- 当前结果类型
- 当前阶段
- 结算后阶段
- 本次新增能力摘要

### 7.6 职业卡候选规则

每次职业卡三选一必须满足：

- 至少一张可以立即提升当前战力
- 至少一张可以补足当前队伍结构
- 不出现三张全部无效

职业卡详细规则统一采用：

- `class_promotion_design.md`

### 7.7 职业卡结算输出

职业卡结算后，统一输出以下结果给 Run 层：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `result_type` | enum | `new_class_unit` / `class_promotion` |
| `class_id` | string | 职业编号 |
| `team_state` | enum | `active` / `bench` |
| `stage_before` | enum | `none` / `low` / `mid` |
| `stage_after` | enum | `low` / `mid` / `high` |
| `summary_key` | string | 本次新增能力摘要键 |

## 8. 队伍系统

### 8.1 队伍构成

- Run 内队伍由上阵单位和候补单位构成
- 上阵人数上限以单场战斗规则为准

### 8.2 候补规则

- 新职业单位在队伍已满时进入候补
- 候补承担补位与冗余职责
- 候补不单独形成第二战斗编组

### 8.3 单位状态跨战保留

Run 层需要保留以下单位状态：

- 存活 / 死亡
- 当前生命
- 当前职业阶段
- 已装备内容
- 是否上阵

### 8.4 Run 持有表

Run 层至少保留以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `class_id` | string | 职业编号 |
| `team_state` | enum | `none` / `active` / `bench` / `dead` |
| `promotion_stage` | enum | `none` / `low` / `mid` / `high` |
| `str` | integer | 力量 |
| `dex` | integer | 敏捷 |
| `con` | integer | 体质 |
| `int` | integer | 智力 |
| `wis` | integer | 感知 |
| `cha` | integer | 魅力 |
| `hp` | integer | 当前生命 |
| `equip_ids` | string[] | 当前装备 |
| `slot_index` | integer | 当前上阵位或空 |

## 9. 装备系统入口

### 9.1 装备来源

Run 内装备来源统一为：

- 商店
- 事件
- 战斗节点掉落

### 9.2 装备职责

- 放大现有 Build
- 提供数值与机制强化
- 补足队伍短板

### 9.3 装备与职业卡分工

- 职业卡负责职业单位获取与进阶
- 装备负责 Run 内强化

---

## 10. 金币系统

### 10.1 金币来源

- 普通战
- 精英战
- 事件
- 章节奖励

### 10.2 金币用途

- 商店购买装备
- 商店购买服务
- 支付事件代价

### 10.3 金币规则

- 金币不替代职业卡
- 金币不直接结算为职业进阶

---

## 11. 招募系统

### 11.1 招募节点职责

- 提供稳定扩编入口
- 提供稳定补位入口
- 提供候补补充入口

### 11.2 招募结算

- 招募节点提供独立招募候选
- 招募候选优先补足当前缺位
- 招募节点不替代战后职业卡三选一

---

## 12. 商店系统

### 12.1 商店内容

商店可出售：

- 装备
- 恢复服务
- 复活服务
- 队伍修复服务

### 12.2 商店规则

- 商店使用金币结算
- 商店不产出职业进阶

---

## 13. 事件系统

### 13.1 事件结果类型

事件可产出以下结果：

- 金币
- 装备
- 招募
- 战斗
- 状态变化

### 13.2 事件代价类型

事件可消耗：

- 金币
- 生命
- 单位状态
- 路线机会

---

## 14. 营地系统

### 14.1 营地职责

- 复活单位
- 深度恢复
- 清理严重负面状态

### 14.2 营地结算

营地可提供：

- 复活 1 名单位
- 整队恢复
- 单位状态修复

---

## 15. 章节完成与 Run 结算

### 15.1 章节完成

- 当前 Boss 战获胜后，章节完成
- 若当前版本只做单章，则 Run 直接结束

### 15.2 Run 失败

- 当前上阵队伍无法继续推进时，本次 Run 失败
- 全队死亡时，本次 Run 失败

### 15.3 Run 结算内容

Run 结算至少展示：

- 通关章节
- 路线记录
- 最终队伍
- 主要职业阶段
- 主要装备
- 失败或胜利结果

---

## 16. Act 1 MVP 范围

Act 1 MVP 统一包含：

- 1 张章节地图
- 2 条主路线
- 7 类节点
- 战斗后固定恢复
- 战斗后职业卡三选一
- 职业卡只产出职业单位与同职业进阶
- 装备由商店、事件和战斗掉落产出

---

## 17. 一页结论

`MiniBattle` 的 Roguelike Run 系统统一为：

```text
章节地图推进
→ 路线选择
→ 节点结算
→ 战斗胜利后固定恢复
→ 职业卡三选一
→ 装备 / 金币 / 招募 / 营地共同修正队伍
→ Boss
→ 章节结算
→ Run 结束
```
