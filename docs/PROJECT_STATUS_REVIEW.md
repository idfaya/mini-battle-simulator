## 4. 问题清单

### 4.1 高优先级问题

#### 问题1: battle_skill.lua 职责混合（**项目定位：可接受**）
- **严重性**: 🟢 低（根据项目定位调整）
- **当前状态**: 1,191行，系统层与机制层职责混合
- **项目定位**: 独立小游戏，<2000行保持单文件

**分析**:

```
battle_skill.lua 结构 (共1,191行):
├─ 系统层职责 (约 650行, 55%) - 技能生命周期、冷却管理、释放控制
├─ 机制层职责 (约 350行, 29%) - 目标选择、条件检查
└─ 工具/配置 (约 191行, 16%)
```

**结论**: 1,191行在可接受范围内，无需强制拆分。如需优化，可用 Lua table 内部组织，不必物理拆分。

**对比参考 - battle_range 拆分**:
- battle_range 拆分三层是出于历史原因和模块复用需求
- skill 模块调用方单一，保持单文件更简洁

#### 问题2: 部分模块仍存在层级越界
- **严重性**: 🟡 中
- **描述**: battle_dmg_heal.lua 直接触发UI层事件
- **代码位置**: battle_dmg_heal.lua 约100行
- **问题**: 机制层直接调用 `BattleEvent.Publish(BattleVisualEvents.DAMAGE_DEALT, ...)`
- **建议**: 机制层只返回计算结果，由系统层统一触发事件

### 4.2 中优先级问题

#### 问题3: 距离计算存在重复实现
- **严重性**: 🟡 中
- **描述**: 距离计算逻辑在多个模块中重复
- **受影响模块**:
  - battle_range.lua: DistanceCalc.Calculate()
  - battle_target_selector.lua: DistanceCalculator.Calculate()
  - battle_area.lua: DistanceCalculator.BetweenUnits()
- **建议**: 统一使用规则层提供的距离计算

#### 问题4: 命名规范不完全统一
- **严重性**: 🟢 低
- **描述**: 部分模块命名不符合层级约定
- **示例**:
  - battle_formation.lua (缺少层级前缀)
  - battle_area.lua (缺少mechanics标识)
- **建议**: 规则层用`{domain}_rules.lua`，机制层用`{domain}_mechanics.lua`

### 4.3 潜在风险

#### 风险1: 循环依赖
- **状态**: 🟡 潜在风险
- **描述**: 暂未形成真正的循环依赖，但依赖关系复杂
- **风险点**:
  - battle_skill.lua 使用函数内require规避循环
  - battle_target_selector.lua 使用pcall包装require
- **建议**: 长期引入依赖注入机制

#### 风险2: battle_range.lua 边界模糊 (已修复)
- **状态**: ✅ 已修复
- **描述**: 原模块同时包含规则层和机制层内容
- **修复状态**: ✅ 已拆分为battle_range_rules.lua和battle_range_mechanics.lua
