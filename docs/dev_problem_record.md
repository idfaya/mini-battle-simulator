# MiniBattleSimulator 开发问题记录 (2026-03-25)

本文档专门记录 MiniBattleSimulator 项目在开发 ViewportRenderer 过程中的技术问题、解决方案及经验总结。

## 1. 终端渲染与字符对齐

### 1.1 UTF-8 字符宽度偏差
- **问题**: 使用边框符（`║`）、血条方块（`█`）和职业图标（`⚔`）时，右边界对齐失效，出现断裂或缩进。
- **原因**: 
  - Lua 的 `#` 操作符计算的是字节数。
  - 边框符和方块虽然是 3 字节 UTF-8，但在终端中只占 1 个显示宽度。
  - 职业图标（如 `⚔`）在不同终端环境下可能被视为 1 宽度或 2 宽度（Emoji）。
- **方案**: 
  - 实现 `GetDisplayWidth` 函数，通过识别字节前缀（如 `0xE2`）并结合特定区间判断宽度。
  - 将边框、方块及特定职业图标统一归类为 **1 宽度**。
- **避坑**: 严禁在 UI 布局计算中直接使用 `#` 长度。

### 1.2 终端原地刷新失效
- **问题**: 画面在 IDE 终端不断向下滚动，无法实现固定视口效果。
- **原因**: ANSI 码 `\27[H` 无法完全清除某些终端的历史缓冲区，且输出行数超过视口高度。
- **方案**: 
  - 使用系统级清屏命令 `os.execute("cls")`。
  - 将 `SCREEN_WIDTH` 限制为 80，`SCREEN_HEIGHT` 限制为 26，适配标准终端。

## 2. 战斗逻辑与性能优化

### 2.1 回合间冗长停顿
- **问题**: 角色行动之间有时会卡住数秒，屏幕无响应。
- **原因**: 行动条系统在计算过程中会产生“空转”帧（没有任何角色进度满），原逻辑每帧都会等待，导致累积延迟。
- **方案**: 在 `BeginNextAction` 中引入 `while` 循环，自动跳过空转帧，直到获取到有效行动角色。

## 3. 视觉表现与阵型

### 3.1 职业图标显示不一致
- **问题**: 敌方角色的职业图标全显示为默认的交叉剑。
- **原因**: `enemy_data.lua` 在生成数据时未将配置中的 `Class` 字段传递给英雄对象。
- **方案**: 确保 `heroData` 对象包含 `class` 属性，并将其映射到 `JOB_ICONS`。

### 3.2 阵型布局错误
- **问题**: 角色位置（wpType）与实际排数对应关系混乱。
- **原因**: 错误地将 `Class`（职业）和 `wpType`（站位）概念混淆，且布局硬编码为两列。
- **方案**: 
  - 重构坐标系，明确 `wpType 1-3` 为前排，`4-6` 为后排。
  - 角色在绘制时根据其 `wpType` 动态查找坐标。
  - 行与行之间增加空行，提升可读性。

### 3.3 视觉反馈增强
- **需求**: 需要快速识别当前行动者，且伤害数字不应遮挡过多画面。
- **方案**: 
  - 当前行动者名字和图标变黄。
  - 伤害数字取消上飘动画，改为直接覆盖在血条中央显示。

## 4. 调试建议
- **1v1 测试**: 使用 `lua test_viewport_battle.lua 50 1 1 500` 快速验证单体渲染和数值显示。
- **满编测试**: 使用 `lua test_viewport_battle.lua 50 6 6 500` 验证阵型对齐和重叠情况。
